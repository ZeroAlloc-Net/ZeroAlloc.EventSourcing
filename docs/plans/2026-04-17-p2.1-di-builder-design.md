# P2.1 — DI Fluent Builder Design

**Date:** 2026-04-17
**Status:** Approved

## Goal

Add a fluent `EventSourcingBuilder` so users can wire up the full event-sourcing stack with a single readable call chain instead of manual `AddSingleton` calls.

## Architecture

`EventSourcingBuilder` lives in the core `ZeroAlloc.EventSourcing` package. Each adapter package (`InMemory`, `PostgreSql`, `SqlServer`, `Sql`, `Kafka`, `Aggregates`) extends it with its own `.Use*()` methods. No new NuGet package is created — this is the same pattern as `IHealthChecksBuilder` in ASP.NET Core.

```csharp
public sealed class EventSourcingBuilder
{
    public IServiceCollection Services { get; }
    internal EventSourcingBuilder(IServiceCollection services) => Services = services;
}
```

`AddEventSourcing()` return type changes from `IServiceCollection` to `EventSourcingBuilder`. This is a breaking change; the method was introduced this session so no external consumers exist.

```csharp
public static EventSourcingBuilder AddEventSourcing(this IServiceCollection services)
{
    services.TryAddSingleton<IEventSerializer, ZeroAllocEventSerializer>();
    return new EventSourcingBuilder(services);
}
```

Users who need to chain back to `IServiceCollection` use `.Services`:

```csharp
services.AddEventSourcing()
        .UsePostgreSqlEventStore(cs)
        .Services
        .AddSingleton<IMyService, MyService>();
```

## Registrations Per Package

All registrations use `TryAddSingleton`. User-provided registrations are never overwritten. `IEventStore` is always composed via DI (`IEventStoreAdapter` + `IEventSerializer` + user-provided `IEventTypeRegistry`).

### `ZeroAlloc.EventSourcing` (core)

| Method | Registers |
|---|---|
| `AddEventSourcing()` | `IEventSerializer → ZeroAllocEventSerializer` |

### `ZeroAlloc.EventSourcing.InMemory`

| Method | Registers |
|---|---|
| `UseInMemoryEventStore()` | `IEventStoreAdapter → InMemoryEventStoreAdapter`, `IEventStore → EventStore` |
| `UseInMemoryCheckpointStore()` | `ICheckpointStore → InMemoryCheckpointStore` |
| `UseInMemorySnapshotStore()` | `ISnapshotStore → InMemorySnapshotStore` |
| `UseInMemoryDeadLetterStore()` | `IDeadLetterStore → InMemoryDeadLetterStore` |
| `UseInMemoryProjectionStore()` | `IProjectionStore → InMemoryProjectionStore` |

## Testing

Each adapter package gets a dedicated test class. Tests are pure unit tests — no containers, no integration fixtures. Pattern:

1. Create `ServiceCollection`
2. Register a stub `IEventTypeRegistry`
3. Call builder chain
4. `BuildServiceProvider().GetRequiredService<T>()` — assert correct concrete type

Each test class covers:
- Each `.Use*()` method resolves the expected type
- `TryAddSingleton` semantics — a prior user registration is not overwritten
- Return is `EventSourcingBuilder` (enables chaining)

Existing `ServiceCollectionExtensionsTests` gets one fix: return-type assertion changes from `IServiceCollection` to `EventSourcingBuilder`.

## DI Dependencies

Each adapter package gains `<PackageReference Include="Microsoft.Extensions.DependencyInjection" />` (already in `Directory.Packages.props` at 10.0.0).
