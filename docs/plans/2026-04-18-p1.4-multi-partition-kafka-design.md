# P1.4 — Multi-Partition Kafka Support Design

**Date:** 2026-04-18

---

## Goal

Replace the single `KafkaStreamConsumer` with two focused implementations — one for explicit partition assignment and one for consumer-group subscription with rebalancing — covering both modes fully with unit and integration tests.

---

## Context

`KafkaConsumerOptions.Partitions` is already `int[]` and per-partition checkpoint tracking (`_lastPositionPerPartition`) already works internally. The gaps are:

- No consumer group subscription / rebalance support
- `GetPositionAsync` returns only the first partition's position
- A single options class mixes manual and group-mode fields
- No unit tests for multi-partition behaviour

---

## Class Structure

### `KafkaConsumerBase` (abstract, internal)

Shared across both modes:

- Poll loop (`ConsumeAsync`)
- Batch processing, retry, dead-letter routing
- `_lastPositionPerPartition : ConcurrentDictionary<int, StreamPosition>`
- `CommitAsync` — writes all tracked partitions to `ICheckpointStore`
- `ResetPositionAsync` — resets all assigned partitions to the given position
- `GetPositionAsync` — returns the minimum position across all tracked partitions; `null` if any partition has no checkpoint yet
- Checkpoint key: `$"{ConsumerId}:p{partition}"`

Two abstract hooks for subclasses:

```csharp
protected abstract Task AssignPartitionsAsync(CancellationToken ct);
protected abstract Task ReleasePartitionsAsync(CancellationToken ct);
```

### `KafkaManualPartitionConsumer : KafkaConsumerBase`

- Reads `KafkaManualPartitionOptions.Partitions`
- On `AssignPartitionsAsync`: reads checkpoint for each partition, seeks consumer, calls `consumer.Assign()`
- On `ReleasePartitionsAsync`: commits all checkpoints, clears tracking

### `KafkaConsumerGroupConsumer : KafkaConsumerBase`

- On startup: registers rebalance handlers, calls `consumer.Subscribe(topic)`
- **Assigned handler** (sync Kafka callback): enqueues newly assigned partitions into `_pendingSeeks : ConcurrentQueue`; returns immediately — no async, no blocking
- **Poll loop** (async): drains `_pendingSeeks` before each `Consume()` call; for each pending partition, `await`s checkpoint read, seeks consumer, adds to `_lastPositionPerPartition`
- **Revoked handler** (sync Kafka callback): calls synchronous `_consumer.Commit()` (Confluent.Kafka sync API — not our async code); removes revoked partitions from `_lastPositionPerPartition`
- After each batch: `await _checkpointStore.WriteAsync(...)` per partition AND `_consumer.StoreOffset(...)` (sync) so Kafka broker offsets are always current for the revoke path
- Empty assignment (zero partitions): poll loop parks without processing until partitions arrive

**No `.GetAwaiter().GetResult()` anywhere.** The revoke callback uses only sync Confluent.Kafka APIs; the assign callback defers all async work to the poll loop.

## Options & DI API

### `KafkaBaseOptions` (abstract)

Common fields: `BootstrapServers`, `Topic`, `ConsumerId`, `PollTimeout`, `ConsumerOptions` (`StreamConsumerOptions`).

### `KafkaManualPartitionOptions : KafkaBaseOptions`

Adds `int[] Partitions` (required, validated non-empty). Optional `GroupId` for Kafka broker-side offset storage.

### `KafkaConsumerGroupOptions : KafkaBaseOptions`

Adds `GroupId` (required). No `Partitions` — Kafka assigns dynamically.

### Builder extensions

```csharp
// Manual assignment
services.AddEventSourcing()
    .UseKafkaManualPartitions(new KafkaManualPartitionOptions
    {
        BootstrapServers = "localhost:9092",
        Topic            = "orders",
        ConsumerId       = "orders-consumer",
        Partitions       = [0, 1, 2],
    });

// Consumer group
services.AddEventSourcing()
    .UseKafkaConsumerGroup(new KafkaConsumerGroupOptions
    {
        BootstrapServers = "localhost:9092",
        Topic            = "orders",
        GroupId          = "orders-group",
        ConsumerId       = "orders-consumer",
    });
```

`KafkaConsumerOptions` and `UseKafka()` are removed (breaking change scoped to the Kafka package only — `IStreamConsumer` contract is unchanged).

## Testing

### Unit tests (mock `IConsumer<string, byte[]>`)

**`KafkaManualPartitionConsumerTests`**
- Startup reads checkpoints for each partition and assigns them
- `ConsumeAsync` processes messages, updates per-partition positions
- `CommitAsync` writes all partition checkpoints
- `GetPositionAsync` returns min across partitions; null if any is unset
- `ResetPositionAsync` resets all partitions to given position

**`KafkaConsumerGroupConsumerTests`**
- `Subscribe` called (not `Assign`) at startup
- Assigned handler enqueues partitions → poll loop seeks to checkpoint positions
- Assigned handler with no prior checkpoint → seeks to `Offset.Beginning`
- Revoked handler calls sync `Commit()` + removes partitions from tracking
- Partitions lost mid-batch: in-progress work for revoked partitions abandoned cleanly
- Empty assignment (zero partitions) → poll loop parks without processing

**`KafkaConsumerBaseTests`**
- Per-partition checkpoint key format `"{ConsumerId}:p{partition}"`
- Batch processing routes events to handler with correct `EventEnvelope`
- Retry policy applied on handler failure
- Dead-letter routing on exhausted retries

### Integration tests (require running Kafka)

- Manual mode: two partitions, events on each → both consumed, per-partition checkpoints written
- Consumer group mode: two consumer instances → partitions split, no duplicate processing
- Rebalance: stop one consumer instance → other picks up its partitions from last checkpoint

### Contract tests

Both `KafkaManualPartitionConsumer` and `KafkaConsumerGroupConsumer` satisfy `IStreamConsumer` contract via a shared base contract test class.

## Breaking Changes

| Removed | Replacement |
|---|---|
| `KafkaConsumerOptions` | `KafkaManualPartitionOptions` / `KafkaConsumerGroupOptions` |
| `KafkaStreamConsumer` | `KafkaManualPartitionConsumer` / `KafkaConsumerGroupConsumer` |
| `UseKafka(options)` | `UseKafkaManualPartitions(options)` / `UseKafkaConsumerGroup(options)` |

`IStreamConsumer`, `ICheckpointStore`, `EventEnvelope` — unchanged.
