# P2.4 + P2.5 — Health Checks & Event Upcasting Design

**Date:** 2026-04-17

---

## P2.4 — Health Checks

### Goal

Add `IHealthCheck` implementations for each adapter so applications can surface event store, checkpoint store, and Kafka broker connectivity in the standard ASP.NET Core health endpoint.

### Approach

Follow the [Xabaril/AspNetCore.Diagnostics.HealthChecks](https://github.com/Xabaril/AspNetCore.Diagnostics.HealthChecks) pattern: one health check class per adapter, registered via `IHealthChecksBuilder` extension methods. No new packages — checks live in the existing adapter packages.

New dependency in each adapter package: `Microsoft.Extensions.Diagnostics.HealthChecks.Abstractions` (interface only, no runtime).

### Registration API

```csharp
// PostgreSQL (ZeroAlloc.EventSourcing.Sql)
services.AddHealthChecks()
    .AddPostgreSqlEventStore("Host=localhost;Database=events")
    .AddPostgreSqlCheckpointStore("Host=localhost;Database=events");

// SQL Server (ZeroAlloc.EventSourcing.SqlServer)
services.AddHealthChecks()
    .AddSqlServerEventStore("Server=.;Database=events;Trusted_Connection=true");

// Kafka (ZeroAlloc.EventSourcing.Kafka)
services.AddHealthChecks()
    .AddKafkaEventStore("localhost:9092");

// InMemory (ZeroAlloc.EventSourcing.InMemory)
services.AddHealthChecks()
    .AddInMemoryEventStore();
```

All methods accept an optional `name` and `tags` parameter matching standard `IHealthChecksBuilder` conventions.

### Check implementations

| Package | Class | Check |
|---|---|---|
| `EventSourcing.Sql` | `PostgreSqlEventStoreHealthCheck` | `SELECT 1` via `NpgsqlDataSource` |
| `EventSourcing.Sql` | `PostgreSqlCheckpointStoreHealthCheck` | `SELECT 1` via `NpgsqlDataSource` |
| `EventSourcing.SqlServer` | `SqlServerEventStoreHealthCheck` | `SELECT 1` via `SqlConnection` |
| `EventSourcing.Kafka` | `KafkaEventStoreHealthCheck` | Metadata fetch from broker |
| `EventSourcing.InMemory` | `InMemoryEventStoreHealthCheck` | Always returns `Healthy` |

**Out of scope:** consumer lag health check — requires Kafka admin client and is a separate operational concern.

### Error handling

- Exceptions caught in `CheckHealthAsync` → `HealthCheckResult.Unhealthy(exception.Message, exception)`
- Timeout via the standard `HealthCheckRegistration.Timeout` — no custom timeout logic

## P2.5 — Event Upcasting

### Goal

Allow events whose payload shape has changed between versions to be transparently upgraded during `IEventStore.ReadAsync` without rewriting historical data.

### Approach

- `IEventUpcaster<TOld, TNew>` — typed user-facing interface
- `IUpcasterPipeline` — internal abstraction used by `EventStore`; walks a registered chain of upcasters
- Auto-composition: registering `v1→v2` + `v2→v3` automatically applies both when reading a `v1` event
- `IEventTypeRegistry` is unchanged — old type names remain registered so old events deserialize to their old CLR type; upcasting is a post-deserialization step
- `IUpcasterPipeline` is optional in `EventStore` — no breaking change; null = no-op

### Abstractions (core package)

```csharp
// User-facing interface
public interface IEventUpcaster<TOld, TNew>
{
    TNew Upcast(TOld oldEvent);
}

// Internal pipeline walking the registered chain
public interface IUpcasterPipeline
{
    // Returns true and sets upgraded if any upcaster applied; false if no upcaster registered for the event type.
    bool TryUpcast(object oldEvent, out object upgraded);
}
```

### Registration

```csharp
services.AddEventSourcing()
    .UseInMemoryEventStore()
    .AddUpcaster<OrderCreatedV1, OrderCreatedV2>(e => new OrderCreatedV2(e.OrderId, "unknown"))
    .AddUpcaster<OrderCreatedV2, OrderCreated>(e => new OrderCreated(e.OrderId, e.CustomerId));
// Reading a stored OrderCreatedV1 event yields an OrderCreated via two hops.
```

`AddUpcaster<TOld, TNew>` registers:
1. `IEventUpcaster<TOld, TNew>` as the typed upcaster in DI
2. An `UpcasterRegistration` descriptor (TOld → TNew type mapping) consumed by the pipeline

### Pipeline implementation (`UpcasterPipeline`)

- Constructed from all registered `UpcasterRegistration` descriptors
- Builds a `Dictionary<Type, Func<object, object>>` at construction time — one entry per registered hop
- `TryUpcast` walks the chain: apply upcaster, check if output type also has an upcaster, repeat until stable
- Max chain depth guard (e.g. 32) to prevent infinite loops from misconfiguration

### `EventStore.ReadAsync` integration

Single change in `ReadAsync` (and `SubscribeAsync`):

```csharp
var deserialized = _serializer.Deserialize(raw.Payload, type);
if (_upcasterPipeline?.TryUpcast(deserialized, out var upgraded) == true)
    deserialized = upgraded;
yield return new EventEnvelope(id, raw.Position, deserialized, raw.Metadata);
```

`_upcasterPipeline` is an optional constructor parameter defaulting to `null`.

### Constraints

- Forward-only: no downcasting
- Idempotent: if no upcaster is registered for a type, the event passes through unchanged
- AOT-safe: no reflection in the hot path — the `Func<object, object>` delegates are built once at startup via registered `IEventUpcaster<TOld, TNew>` instances
