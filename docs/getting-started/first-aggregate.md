# Your First Aggregate

Build your first event-sourced aggregate in 5 minutes.

## What You'll Build

An `Order` aggregate that demonstrates the complete event-sourcing lifecycle:
- Places orders with a total amount
- Ships orders with tracking information
- Applies events to maintain aggregate state
- Persists to and loads from an event store

This tutorial introduces the fundamental patterns: defining events, maintaining state through event application, and reconstructing aggregates from their event history.

## Step 1: Define the Order ID and Events

Events represent things that happened in your domain. They are immutable records of facts that can never be undone—only superseded by newer events. Start by defining the aggregate ID type and events.

The aggregate ID is a value type that uniquely identifies your aggregate. Events are domain-focused records that capture the "what" of a business operation without specifying how to handle them.

```csharp
using ZeroAlloc.EventSourcing;

namespace MyProject.Domain;

// Define the aggregate ID as a value type
// Value types are preferred for aggregate IDs: they're allocation-free and hashable
public readonly record struct OrderId(Guid Value);

// Events are immutable records describing what happened
// Each event should capture domain-significant information
// Note: Events use strings for domain data (e.g., "ORD-001" as a user-facing order number)
// while the aggregate ID (OrderId) is a Guid for internal uniqueness and efficiency.
// This separation allows the domain model to use human-friendly values in events
// while using allocation-free value types for aggregate identity.
public record OrderPlacedEvent(string OrderId, decimal Total);

public record OrderShippedEvent(string TrackingNumber);
```

## Step 2: Define State

State is the current condition of your aggregate, derived from the history of events that have occurred. In event sourcing, state is not stored directly—only events are. State is reconstructed by replaying events.

The state type must implement `IAggregateState<T>`, which requires an `Initial` property defining the empty state. The state type should be a `struct` to avoid heap allocations during event replay. Each event applies to the state and produces a new state via immutable updates (using the `with` keyword for records).

```csharp
namespace MyProject.Domain;

// State is a struct for allocation-free event replay
public partial struct OrderState : IAggregateState<OrderState>
{
    // Required: Define the initial empty state
    // Structs default to all fields zero, so 'default' represents a new order
    public static OrderState Initial => default;
    
    // State properties capture the current condition
    public bool IsPlaced { get; private set; }
    public decimal Total { get; private set; }
    public string TrackingNumber { get; private set; }

    // Apply events to produce new state
    // Each method is pure: it takes the current state and event, returns new state
    internal OrderState Apply(OrderPlacedEvent e) =>
        this with { IsPlaced = true, Total = e.Total };

    internal OrderState Apply(OrderShippedEvent e) =>
        this with { TrackingNumber = e.TrackingNumber };
}
```

## Step 3: Define Your Aggregate

The aggregate is the command handler in your domain model. It inherits from `Aggregate<TId, TState>` and encapsulates domain behavior—the rules about what can happen and when.

Aggregates never mutate state directly. Instead, they use `Raise()` to emit domain events, which are applied to state immediately. This ensures:
- State is always consistent (never in an invalid intermediate state)
- All mutations are auditable (every state change has a corresponding event)
- Aggregates can be replayed from history by applying the same events

The `ApplyEvent` method is a dispatcher that routes each event type to the correct state apply method. This method is typically generated by a source generator (ZeroAlloc.EventSourcing.Generators) for `partial` aggregate classes.

```csharp
using ZeroAlloc.EventSourcing;

namespace MyProject.Domain;

public sealed partial class Order : Aggregate<OrderId, OrderState>
{
    // Domain behavior: Place an order
    // Uses Raise() to emit an event and apply it to state atomically
    public void Place(string orderId, decimal total)
    {
        Raise(new OrderPlacedEvent(orderId, total));
    }

    // Domain behavior: Ship an order
    // Domain rules are enforced before raising events
    public void Ship(string trackingNumber)
    {
        if (!State.IsPlaced)
            throw new InvalidOperationException("Cannot ship an order that hasn't been placed");
        
        Raise(new OrderShippedEvent(trackingNumber));
    }

    // Set the aggregate ID (called after loading or during creation)
    public void SetId(OrderId id) => Id = id;

    // Apply events to state - returns new state
    // This dispatcher routes events to their corresponding Apply methods on OrderState
    protected override OrderState ApplyEvent(OrderState state, object @event) => @event switch
    {
        OrderPlacedEvent e => state.Apply(e),
        OrderShippedEvent e => state.Apply(e),
        _ => state
    };
}
```

## Step 4: Save and Load

Now that you have an aggregate, you can persist its events to a store and later reconstruct it from history.

### Saving Events

When you call `Raise()` on an aggregate, events are queued in memory but not yet persisted. Call `DequeueUncommitted()` to get the events and prepare them for storage. The event store persists these events in a stream identified by the aggregate's ID.

### Loading Events

Event stores return events wrapped in `EventEnvelope` structures, which contain both the event payload and its `StreamPosition`. The position is critical: it identifies where the event sits in the stream's total ordering and is needed for optimistic concurrency control during saves.

When reconstructing an aggregate, you call `ApplyHistoric()` with both the event and its position. This applies the event to state and updates the aggregate's internal version tracking.

```csharp
using ZeroAlloc.EventSourcing;

// === Create and modify an aggregate ===
var orderId = new OrderId(Guid.NewGuid());
var order = new Order();
order.SetId(orderId);
order.Place("ORD-001", 1500m);
order.Ship("TRACK-12345");

// Get uncommitted events
// After calling Place() and Ship(), the aggregate has two uncommitted events
var events = order.DequeueUncommitted();
// events now contains: [OrderPlacedEvent, OrderShippedEvent]

// === Persist events to store ===
// EventStore requires three components:
// 1. Adapter (InMemoryEventStoreAdapter for in-memory, or custom for databases)
// 2. Serializer (typically application-specific, handles event serialization)
// 3. Registry (usually source-generated, maps event types for serialization)
var adapter = new InMemoryEventStoreAdapter();
var registry = new OrderEventTypeRegistry();    // source-generated from your events
var serializer = new JsonAggregateSerializer(); // application-provided serializer
var eventStore = new EventStore(adapter, serializer, registry);

var streamId = StreamId.From(orderId);
// Append to the event store at the expected version
// StreamPosition.Start means "start of stream" (first append should be at Start)
await eventStore.AppendAsync(streamId, events, StreamPosition.Start);

// === Load aggregate from event store ===
// Create a new aggregate instance to be reconstructed
var loadedOrder = new Order();
loadedOrder.SetId(orderId);

// Read all events from the stream
// ReadAsync returns EventEnvelope structures containing Event and Position
await foreach (var envelope in eventStore.ReadAsync(streamId, StreamPosition.Start))
{
    // ApplyHistoric applies the event and updates internal version tracking
    // It requires both the event payload and its position in the stream
    loadedOrder.ApplyHistoric(envelope.Event, envelope.Position);
}

// State is now fully reconstructed from history
// The loadedOrder aggregate is an exact replica of the original 'order' aggregate
Console.WriteLine($"Order IsPlaced: {loadedOrder.State.IsPlaced}");    // true
Console.WriteLine($"Order Total: {loadedOrder.State.Total}");          // 1500
Console.WriteLine($"Tracking: {loadedOrder.State.TrackingNumber}");     // TRACK-12345
```

## Key Concepts

### Event Envelope
When reading from an event store, events are returned inside an `EventEnvelope` structure:
- `Event`: The event payload (e.g., `OrderPlacedEvent`)
- `Position`: The stream position where the event is located (for versioning)
- `StreamId`: The stream identifier
- `Metadata`: Additional event metadata

The envelope is necessary because event position information is essential for optimistic concurrency control. When saving events, the event store compares the expected version against the actual stream version to detect conflicts.

### Aggregate Versioning
Each aggregate tracks two versions:
- `Version`: The current version (incremented each time an event is applied)
- `OriginalVersion`: The version when the aggregate was loaded from the store

When saving, the event store uses `OriginalVersion` to ensure no other process has modified the stream concurrently. This is optimistic concurrency control—conflicts are detected at save time rather than preventing access.

### Allocation Efficiency
The library uses value types (`struct`) for both state and aggregate IDs to minimize heap allocations during event replay. When replaying thousands of events, avoiding allocations per event is critical for performance. This zero-allocation design is what makes ZeroAlloc.EventSourcing suitable for high-throughput systems.

You've now built a complete event-sourced aggregate: defining domain events, maintaining state through immutable transitions, and persisting/reconstructing from a history of events. This pattern scales from simple domains to complex enterprise systems.
